Gap analysis observations to review — not assertions of bugs

watch Hub referential drift: vehicle hub vs. deployment hub

Both vehicles.hub_id (0014) and deployments.hub_id exist independently, with no constraint tying them together. A deployment can record a hub different from where its vehicle is stationed. Decide which is authoritative, or add a check / default deployments.hub_id from the vehicle.

watch Retired hubs are soft-deleted but still FK-referenced

0015 marks HYD/SEC/etc. deleted_at but they remain valid FK targets for existing deployments/vehicles (no cascade, by intent). deployments_enriched joins hubs without a deleted_at IS NULL filter, so retired hub names still render — fine for history, but any "list active hubs" dropdown must filter deleted_at IS NULL or staff can deploy to a dead hub. Worth confirming the new-deployment form does this.

review No DB enforcement that app_users.hub_id is set for non-CMD

The role_hub_check constraint was dropped in 0012 (it fought the signup trigger). The invariant "field staff must have a hub" now lives only in RLS + app logic. An un-hubbed FIELD_STAFF is harmless (RLS filters them to nothing) but also invisible/non-functional until an admin assigns a hub — and the admin UI to do that (Module 9) isn't built yet. Operational gap, not a schema bug.

review payment_mode is loosely coupled to event type

0016 adds payment_mode with a value CHECK but no link to event_type — it can be set on a LOCK or RETURN row, or left NULL on a PAYMENT. If you need clean payment-mode reporting, consider it NOT NULL for money events (enforced in logActivityEvent() or a CHECK).

review Deposit cash vs. deposit contract can diverge silently

deployments.deposit_required_inr + new_deposit_needed describe the contract; actual cash is the sum of DEPOSIT/DEPOSIT_REFUND rows in activity_log. Nothing reconciles the two — a deployment can require ₹X deposit with ₹0 ever collected and no flag. The dashboard may want a "deposit shortfall" signal.

review week_number / extra_weeks not validated against the contract

A PAYMENT can carry any week_number (e.g. week 99 on a 4-week deployment); EXTENSION's extra_weeks only checks > 0, not an upper bound that keeps weeks ≤ 52. The weeks BETWEEN 1 AND 52 CHECK lives on deployments, so a large extension that bumps weeks past 52 would fail at update time — verify logActivityEvent() surfaces that cleanly.

note Two external ID systems on riders, neither required

legacy_rider_id (CSV) and app_rider_id (mobile) are both nullable text with no overlap rule. Fine today, but if the CSV import lands, confirm whether a rider can/should have both and which one search keys off.

note Replacement event mutates deployments.vehicle_id — uniqueness window

REPLACEMENT swaps vehicle_id on the live ACTIVE deployment. The deployments_active_vehicle_uniq partial index guards against double-assigning the new vehicle, but the old vehicle becomes available the instant the row updates. Confirm logActivityEvent() does the swap in one transaction so there's no window where both vehicles look occupied (or neither).

note Soft-delete columns exist everywhere; no delete UI exists yet

Every core table has deleted_at and all query helpers filter it, but there is no Server Action that sets it. Deletes are currently impossible from the UI — intended for now, but flagged so it's a conscious gap, not an oversight.

note created_by/updated_by point at auth.users, not app_users

Audit columns reference the Supabase auth table directly. To show "edited by <name, role, hub>" you must join through app_users separately — no single FK gives you the app-level actor. Minor, but relevant if you build an audit/history screen.
